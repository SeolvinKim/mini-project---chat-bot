# 통합 PRD — 취업준비 도움 챗봇

> 금융권·IT·디지털 분야 취업을 준비하는 사용자를 위한 Gradio 기반 대화형 챗봇  
> 직무 추천, 자소서 피드백, 면접 질문 생성, 자격증 추천·시험 일정 안내의 4개 Tool을 하나의 서비스로 제공한다.

## 1. 제품 개요

### 1.1 한 줄 소개

사용자의 학력, 희망 직무, 기술, 경험과 보유 자격증을 바탕으로 직무 탐색부터 서류·면접·자격증 준비까지 한 대화 안에서 지원하는 통합 취업준비 챗봇.

### 1.2 문제 정의

취업준비생은 취업 과정의 단계마다 서로 다른 정보를 찾아야 한다.

- 자신에게 적합한 직무를 판단하기 어렵다.
- 자소서의 문제점을 객관적으로 확인하기 어렵다.
- 실제 면접에서 받을 질문과 답변 방향을 준비하기 어렵다.
- 직무에 필요한 자격증과 시험 일정이 여러 사이트에 흩어져 있다.
- 여러 서비스를 오가며 같은 프로필을 반복해서 입력해야 한다.

본 서비스는 하나의 사용자 프로필과 대화형 UI를 이용해 네 가지 취업 준비 기능을 연결한다.

### 1.3 제품 목표

1. 사용자가 자신의 현재 상태에 맞는 취업 준비 기능을 쉽게 선택할 수 있게 한다.
2. 한 번 입력한 프로필을 네 개 Tool이 공통으로 활용하게 한다.
3. 각 Tool이 사용자가 바로 실행할 수 있는 구체적인 결과를 제공하게 한다.
4. RAG와 LLM을 활용하되 확인되지 않은 사실은 생성하지 않게 한다.
5. 개별 Tool 장애가 전체 챗봇 중단으로 이어지지 않게 한다.
6. GitHub에서 네 명의 작업을 충돌 없이 통합하고 Render에 배포한다.

### 1.4 MVP 범위

| 기능 | 핵심 제공 가치 |
|---|---|
| 기능 1. 직무 추천 | 사용자 프로필에 적합한 금융권 직무 상위 3개 추천 |
| 기능 2. 자소서 피드백 | 자소서를 6개 평가 항목으로 진단하고 수정 방향 제공 |
| 기능 3. 면접 질문 생성 | 직무와 지원 자료에 맞는 면접 질문·의도·답변 가이드 생성 |
| 기능 4. 자격증 추천 | 직무 관련 자격증 추천 및 올해 시험 일정 안내 |

### 1.5 MVP 비범위

- 채용 지원서 실제 제출
- 채용공고 자동 지원
- 자소서 파일 편집·저장
- 음성 면접 및 영상 분석
- 합격 가능성 보장 또는 예측
- 자격시험 접수·결제
- 시험장·응시료·합격 여부 조회
- 자격증 이외의 별도 활동 추천
- 사용자 계정과 장기 프로필 저장

## 2. 대상 사용자

### 2.1 핵심 타깃

- 금융권 취업을 준비하는 20~30대
- 금융 IT·핀테크·데이터 직무 취업을 준비하는 20~30대
- 신입 취업, 이직 또는 직무 전환을 준비하는 사용자
- 직무 선정부터 면접 준비까지 체계적인 흐름이 필요한 사용자

### 2.2 대표 사용자 시나리오

#### 시나리오 A — 직무를 정하지 못한 사용자

1. 프로필에 전공, 기술과 관심 분야를 입력한다.
2. 직무 추천 Tool에서 적합한 직무 3개를 확인한다.
3. 추천 직무 중 하나를 희망 직무로 선택한다.
4. 자격증 Tool에서 해당 직무에 필요한 자격증을 확인한다.

#### 시나리오 B — 지원 직무가 정해진 사용자

1. 희망 직무와 경험을 프로필에 입력한다.
2. 자소서 본문을 입력해 항목별 피드백을 받는다.
3. 수정한 자소서와 프로젝트 경험을 기준으로 면접 질문을 생성한다.
4. 부족한 역량을 보완할 자격증과 시험 일정을 확인한다.

#### 시나리오 C — 면접을 앞둔 사용자

1. 지원 기업, 직무, 자소서와 프로젝트 내용을 입력한다.
2. 직무·프로젝트·기업 맞춤 질문을 받는다.
3. 답변 연습 후 추가 또는 압박 질문을 요청한다.

## 3. 공통 사용자 프로필

네 개 Tool은 온보딩에서 생성한 하나의 `UserProfile`을 읽기 전용으로 공유한다.

```python
from pydantic import BaseModel, Field

class UserProfile(BaseModel):
    session_id: str = ""
    education: str = ""
    target_job: str = ""
    skills: list[str] = Field(default_factory=list)
    experiences: list[str] = Field(default_factory=list)
    certs: list[str] = Field(default_factory=list)
```

| 필드 | 설명 | 활용 Tool |
|---|---|---|
| `session_id` | 브라우저 세션 고유 ID — Tool 상태 키잉에 사용 | 자격증 추천 (추천 이력 격리) |
| `education` | 학력·전공 | 직무 추천, 자소서, 면접 |
| `target_job` | 희망 직무 | 네 개 Tool 전체 |
| `skills` | 보유 기술 | 직무 추천, 자소서, 면접, 자격증 |
| `experiences` | 프로젝트·활동 | 직무 추천, 자소서, 면접 |
| `certs` | 보유 자격증 | 직무 추천, 자격증 |

### 공통 규칙

- Tool은 `profile`을 읽기만 하고 수정하지 않는다.
- 프로필에 없는 추가 정보는 현재 대화의 `user_input`으로 받는다.
- 자소서 본문처럼 긴 입력은 프로필에 저장하지 않고 Tool 요청으로 전달한다.
- 사용자가 프로필을 변경하면 이후 요청부터 변경된 프로필을 사용한다.

## 4. 공통 Tool 계약

각 Tool 파일은 `NAME`과 `run()`만 외부에 노출한다.

```python
from core.schema import UserProfile

NAME = "화면 표시 이름"

def run(profile: UserProfile, user_input: str) -> str:
    ...
```

### Tool 목록

| 기능 | 파일 | `NAME` | RAG | Chroma 컬렉션 |
|---|---|---|:---:|---|
| 직무 추천 | `tools/recommend_job.py` | `직무 추천` | 미사용 | 없음 (오프라인 규칙 기반) |
| 자소서 피드백 | `tools/cover_letter.py` | `자소서 피드백` | 미사용 | 없음 |
| 면접 질문 | `tools/interview.py` | `면접 질문` | 미사용 | 없음 |
| 자격증 추천 | `tools/spec_recommend.py` | `자격증 추천` | 선택적 사용 | `certs` (없으면 JSON 기반으로 폴백) |

### 반환 규칙

- 모든 Tool은 사람이 읽을 수 있는 문자열 하나를 반환한다.
- `dict`, `list`, 객체 또는 `None`을 반환하지 않는다.
- Markdown 표와 목록은 문자열 안에 포함할 수 있다.
- 예외가 발생해도 사용자에게 이해 가능한 문자열을 반환한다.
- OpenAI API 키와 내부 오류 상세를 화면에 노출하지 않는다.

## 5. 공통 사용자 흐름 및 라우팅

### 5.1 전체 사용자 흐름

```text
서비스 진입
  → 온보딩 프로필 입력
  → 기능 선택
      ├─ 직무 추천
      ├─ 자소서 피드백
      ├─ 면접 질문 생성
      └─ 자격증 추천·일정
  → Tool 결과 확인
  → 후속 질문 또는 다른 기능 이동
```

### 5.2 기능 선택 라우팅

| 사용자 의도·입력 | 연결 Tool |
|---|---|
| 적성, 직무, 진로, 어떤 일을 할지 질문 | 직무 추천 |
| 자소서, 지원동기, 문장 수정, 첨삭 요청 | 자소서 피드백 |
| 면접, 예상 질문, 압박 질문, 답변 연습 | 면접 질문 생성 |
| 자격증, 시험 일정, SQLD, 투운사 등 | 자격증 추천 |

- 사용자가 화면 메뉴에서 Tool을 직접 선택할 수 있어야 한다.
- 메뉴를 선택하지 않은 자유 입력은 키워드와 LLM 의도 분류로 Tool을 결정할 수 있다.
- 의도가 불명확하면 임의 실행하지 않고 원하는 기능을 다시 질문한다.
- 한 Tool의 대화 중 다른 기능을 명시하면 해당 Tool로 전환한다.

### 5.3 화면 구성

#### Gradio UI (`app/main.py`)

| 영역 | 구성 요소 | 역할 |
|---|---|---|
| 헤더 | 서비스명, 한 줄 설명 | 서비스 목적 안내 |
| 온보딩 | 학력, 직무, 기술, 경험, 자격증 | 공통 프로필 생성 |
| 기능 메뉴 | 4개 Tool 선택 버튼 | 명시적 라우팅 |
| 채팅 영역 | `gr.Chatbot` | 사용자·챗봇 대화 표시 |
| 입력 영역 | `gr.Textbox`, 전송 버튼 | 자연어 요청 입력 |
| 빠른 질문 | Tool별 예시 버튼 | 초기 사용 지원 |
| 상태 영역 | 선택 Tool, 처리 중, 오류 | 현재 상태 안내 |
| 하단 | 데이터·AI 안내 | 결과 검토 및 공식 확인 고지 |

#### vtuber 프론트엔드 (`vtuber/`)

| 영역 | 구성 요소 | 역할 |
|---|---|---|
| 왼쪽 `#stage` | Three.js VRM 캐릭터 캔버스 | 3D 캐릭터 표시, 립싱크, 아이들 모션 |
| 오른쪽 `#chat-panel` (360px) | 채팅 UI | 메시지 입력·출력, Tool 선택, 음성 선택 |
| 링크 배경 | `#link-frame` iframe | 봇 응답에 URL이 있으면 캔버스 뒤에 표시 |

#### FastAPI HTTP API (`app/api.py`, 포트 8000)

| 엔드포인트 | 역할 |
|---|---|
| `GET /api/health` | 서버 상태 확인 |
| `GET /api/tools` | Tool 목록 (key, label, icon) |
| `GET /api/voices` | TTS 음성 후보 목록 |
| `POST /api/chat` | 메시지 → Tool 라우팅 → 텍스트 응답 |
| `POST /api/tts` | 텍스트 → mp3 음성 합성 |

### 5.4 대화 상태

- Gradio `State`에 현재 Tool, 프로필, 대화 이력과 Tool별 임시 상태를 저장한다.
- 새 대화 시 대화 이력과 Tool 임시 상태를 초기화한다.
- 프로필 초기화는 별도 버튼으로 처리한다.
- 새로고침 이후 영구 상태 보존은 MVP 범위에 포함하지 않는다.

## 6. 기능 1 — 금융권 직무 추천

### 6.1 기능 요구사항

- 자연어 프로필을 바탕으로 금융권 직무 20개 중 상위 3개를 추천한다.
- 각 결과에 추천 이유, 필요한 역량과 준비 방향을 제공한다.
- 입력 정보가 부족하면 추가 질문을 반환한다.
- OpenAI API 없이 오프라인 규칙 기반으로 동작한다.

### 6.2 입력값

- 전공
- 관심 금융 분야
- 보유 경험
- 보유 역량
- 보유 자격증
- 선호 업무 방식

### 6.3 출력값

```text
현재 입력 기준으로는 다음 3개 직무가 잘 맞을 수 있습니다.

1. 직무명
- 추천 이유:
- 필요한 역량:
- 준비 방향:

2. 직무명
- 추천 이유:
- 필요한 역량:
- 준비 방향:

3. 직무명
- 추천 이유:
- 필요한 역량:
- 준비 방향:

마무리 조언:
```

## 10. RAG·LLM·데이터 설계

### 10.1 기술별 역할

| 기술 | 적용 기능 | 역할 |
|---|---|---|
| OpenAI Chat API | 전체 | 의도 파악과 자연어 결과 생성 |
| OpenAI Embeddings | 기능 1·3·4 | 문서 벡터화 |
| LangChain | 전체 | 프롬프트, LLM과 RAG 연결 |
| Chroma | 기능 1·3·4 | 직무·면접·자격증 문서 검색 |
| JSON | 기능 1·3·4 | 수집 원본과 구조화 데이터 저장 |
| Python 규칙 로직 | 전체 | 점수 계산, 검증과 폴백 |

### 10.2 컬렉션

| 컬렉션 | 데이터 |
|---|---|
| `jobs` | 금융권 직무 설명과 요구 역량 |
| `interview_qs` | 유형별·직무별 면접 질문 |
| `certs` | 자격증 설명, 관련 직무와 추천 근거 |

### 10.3 수집 원칙

- 크롤링은 `ingest/`의 오프라인 1회성 스크립트로 수행한다.
- 배포 앱에는 크롤러를 넣지 않는다.
- 수집 원본은 `data/raw/`에 JSON으로 저장한다.
- robots.txt, 접근 제한과 공식 사이트 이용정책을 준수한다.
- 크롤링이 어려우면 소량의 검증된 데이터를 직접 작성한다.
- 배포 앱은 미리 생성한 JSON과 Chroma 컬렉션만 조회한다.

### 10.4 환각 방지

- 검색 근거가 없으면 사실처럼 단정하지 않는다.
- 기업 맞춤 질문은 입력되거나 검색된 기업 정보 범위 안에서 생성한다.
- 자격시험 일정은 JSON 값만 사용한다.
- 사용자의 자소서에 없는 경험이나 성과를 새로 만들지 않는다.
- 결과에 필요한 정보가 없으면 추가 질문을 한다.

### 10.5 장애 폴백

| 장애 | 처리 |
|---|---|
| OpenAI API 실패 | 규칙 기반 안내 또는 재시도 메시지 |

| `interview_qs` 검색 실패 | 기본 유형별 질문 템플릿 |
| `certs` 검색 실패 | JSON 직무 매핑 기반 추천 |
| 시험 일정 데이터 없음 | 공식 링크와 데이터 없음 안내 |

## 13. Render 배포

### 13.1 기본 설정

| 항목 | 값 |
|---|---|
| Service | Render Web Service |
| 연결 저장소 | GitHub 프로젝트 저장소 |
| 배포 브랜치 | `main` |
| Build Command | `uv sync --frozen` |
| Start Command (Gradio) | `uv run python app/main.py` |
| Start Command (FastAPI) | `uv run uvicorn app.api:app --host 0.0.0.0 --port 8000` |
| Host | `0.0.0.0` |
| Port | Render의 `PORT` 환경변수 |

```python
import os

port = int(os.getenv("PORT", "7860"))
demo.launch(server_name="0.0.0.0", server_port=port)
```

### 13.2 환경변수

| 변수 | 설명 |
|---|---|
| `OPENAI_API_KEY` | OpenAI Chat·TTS 인증 |
| `CHAT_MODEL` | LLM 모델 ID (기본 `gpt-4o-mini`) |
| `TTS_PROVIDER` | TTS 우선 순위: `azure`(기본) 또는 `openai` |
| `TTS_MODEL` | OpenAI TTS 모델 (기본 `tts-1`) |
| `TTS_VOICE` | OpenAI TTS 음성 (기본 `alloy`) |
| `AZURE_SPEECH_KEY` | Azure Speech 서비스 키 |
| `AZURE_SPEECH_REGION` | Azure Speech 리전 (기본 `koreacentral`) |
| `AZURE_TTS_VOICE` | Azure TTS 음성 (기본 `ko-KR-SunHiNeural`) |
| `AZURE_TTS_RATE` | Azure TTS 말속도 (기본 `-12%`) |
| `AZURE_TTS_PITCH` | Azure TTS 음높이 (기본 `-8%`) |
| `AZURE_TTS_STYLE` | Azure TTS 스타일 (미지원 음성 적용 시 합성 실패 — 기본 비활성) |
| `PORT` | Render 제공 포트 |
| `APP_ENV` | 개발·운영 환경 구분 |
| `LOG_LEVEL` | 로그 수준 |

### 13.3 배포 데이터

- 실행에 필요한 `chroma_db/`와 JSON 데이터를 배포에 포함한다.
- 배포 시점 또는 런타임에 크롤링하지 않는다.
- 필수 데이터가 없으면 앱 전체가 종료되지 않게 한다.
- Render의 로컬 파일이 영구 저장된다고 가정하지 않는다.

## 14. 비기능 요구사항

- 일반 응답은 5초 이내 표시를 목표로 한다.
- 로컬 데이터 조회는 2초 이내 표시를 목표로 한다.
- Tool 하나의 오류가 전체 앱을 중단시키지 않아야 한다.
- 최신 Chrome·Edge의 PC와 모바일에서 동작해야 한다.
- OpenAI 키, 사용자 프로필과 자소서 원문을 로그에 노출하지 않는다.
- 사용자 입력은 해당 세션에서만 사용한다.
- 모든 Tool은 `uv run` 환경에서 import와 실행이 가능해야 한다.
- 출력은 한국어를 기본으로 한다.

## 15. 테스트 요구사항

### 15.1 Tool 계약 테스트

- 네 개 Tool의 `NAME`, `run` import 성공
- 더미 `UserProfile`로 호출 시 문자열 반환
- 프로필 객체를 수정하지 않음
- 오류 상황에서도 문자열 안내 반환

### 15.2 기능 테스트

| 기능 | 대표 입력 | 기대 결과 |
|---|---|---|
| 직무 추천 | `SQL과 Python을 배우고 금융 데이터 분석에 관심 있어요` | 금융 데이터 분석 중심 직무 3개 |
| 자소서 피드백 | 지원 직무, 문항과 100자 이상 본문 | 6개 항목 기반 피드백 |
| 면접 질문 | 기업, 직무와 프로젝트 입력 | 질문·의도·평가 포인트·가이드 |
| 자격증 추천 | `데이터 분석가 자격증 추천해줘` | 관련 자격증 최대 3개 |
| 자격증 일정 | `SQLD 올해 일정 알려줘` | 예정·종료 일정과 공식 출처 |

### 15.3 통합 테스트

1. 온보딩 프로필 저장
2. 각 기능 메뉴 이동
3. 네 개 Tool 각각 1회 이상 실행
4. 같은 프로필이 네 Tool에 전달되는지 확인
5. Tool 전환 후 이전 Tool의 임시 상태가 섞이지 않는지 확인
6. 새 대화와 프로필 초기화 확인
7. OpenAI 장애 폴백 확인
8. 모바일 UI 확인
9. Render 공개 URL에서 전체 흐름 확인

## 16. 완료 기준

- Gradio 온보딩에서 `UserProfile` 5개 필드를 입력할 수 있다.
- 사용자가 네 개 기능을 선택하고 자연어로 요청할 수 있다.
- 모든 Tool이 공통 함수 계약을 지킨다.
- 직무 추천은 7개 후보 중 상위 3개를 제공한다.
- 자소서 피드백은 6개 평가 항목을 다룬다.
- 면접 질문은 5개 유형을 지원한다.
- 자격증 기능은 추천과 올해 일정 조회를 제공한다.
- 기능 1·3·4의 Chroma 컬렉션 검색이 동작한다.
- OpenAI 또는 RAG 실패 시 사용자 안내나 폴백이 동작한다.
- GitHub PR을 통해 네 기능이 통합된다.
- Render 공개 URL에서 네 기능이 모두 동작한다.
- 비밀키가 저장소와 화면에 노출되지 않는다.
